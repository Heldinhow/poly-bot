Arquitetura Técnica

O que precisa mudar no backend e frontend

1. Backend: Nova rota WebSocket para steps

Endpoint GET /api/executions/{execution_log_id}/stream que transmite cada step do agent em tempo real.

EventBus ──publish──▶ /api/executions/{id}/stream ──WS──▶ Frontend
    │
    │  ExecutionTracker.feed_step()
    │  ├── Persiste no DB (execution_steps)
    │  └── Publica no EventBus (para WS broadcast)
    │
Agent Backend (claude/opencode/etc)
    │
    └── yield Message(type, content, tool, ...)
         ├── thinking  → step_type="thinking"
         ├── tool_use  → step_type="tool_use"
         ├── tool_result → step_type="tool_result"
         ├── text       → step_type="text"
         └── error      → step_type="error"

2. Backend: Modificações no ExecutionTracker

Hoje o tracker persiste steps no DB. Vamos adicionar broadcast via EventBus para cada step.

Antes
feed_step() → INSERT no DB
(silencioso, sem broadcast)
Depois
feed_step() → INSERT no DB
+ event_bus.publish(STEP_EVENT)
→ WS broadcast para frontend

3. Frontend: Novos componentes

Componente Função Baseado em
LiveExecutionCard Substitui ExecutionCard. Header com status/tempo/tools, timeline colorida, steps em tempo real, output final. multica AgentLiveCard
ExecutionTimeline Lista scrollable de steps com badges coloridos por tipo, expansão de detalhes, auto-scroll. multica SingleAgentLiveCard
TimelineProgressBar Barra horizontal com segmentos coloridos proporcionais à contagem de cada tipo de step. multica AgentTranscriptDialog
ExecutionSummary Grid com probabilidade, confiança, edge, decisão + reasoning text. Só aparece quando completed. novo
useExecutionStream Hook que conecta ao WebSocket de steps e gerencia o buffer de eventos + auto-scroll. multica useWSEvent

4. Fluxo de dados frontend

ExecutionsPage
  ├── useExecutions() — polling /api/executions (lista base, a cada 5s)
  │
  └── LiveExecutionCard (para cada execution)
        ├── Estado: queued/claimed → mostra "Aguardando..."
        ├── Estado: running → conecta WebSocket
        │     ├── useExecutionStream(executionId)
        │     │     ├── ws://.../api/executions/{id}/stream
        │     │     ├── Buffer de steps em estado local
        │     │     └── Auto-scroll com detecção de scroll-up
        │     ├── TimelineProgressBar (segmentos coloridos)
        │     └── ExecutionTimeline (steps em tempo real)
        │
        └── Estado: completed/failed → mostra output final
              ├── ExecutionSummary (prob, conf, edge, decisão + reasoning)
              ├── TimelineProgressBar (estático, completo)
              └── ExecutionTimeline (colapsado, expansível)
